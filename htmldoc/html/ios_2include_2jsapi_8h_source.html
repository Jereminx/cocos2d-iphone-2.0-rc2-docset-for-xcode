00047 /*
00048  * Protecting non-jsval, non-JSObject *, non-JSString * values from collection
00049  *
00050  * Most of the time, the garbage collector's conservative stack scanner works
00051  * behind the scenes, finding all live values and protecting them from being
00052  * collected. However, when JSAPI client code obtains a pointer to data the
00053  * scanner does not know about, owned by an object the scanner does know about,
00054  * Care Must Be Taken.
00055  *
00056  * The scanner recognizes only a select set of types: pointers to JSObjects and
00057  * similar things (JSFunctions, and so on), pointers to JSStrings, and jsvals.
00058  * So while the scanner finds all live |JSString| pointers, it does not notice
00059  * |jschar| pointers.
00060  *
00061  * So suppose we have:
00062  *
00063  *   void f(JSString *str) {
00064  *     const jschar *ch = JS_GetStringCharsZ(str);
00065  *     ... do stuff with ch, but no uses of str ...;
00066  *   }
00067  *
00068  * After the call to |JS_GetStringCharsZ|, there are no further uses of
00069  * |str|, which means that the compiler is within its rights to not store
00070  * it anywhere. But because the stack scanner will not notice |ch|, there
00071  * is no longer any live value in this frame that would keep the string
00072  * alive. If |str| is the last reference to that |JSString|, and the
00073  * collector runs while we are using |ch|, the string's array of |jschar|s
00074  * may be freed out from under us.
00075  *
00076  * Note that there is only an issue when 1) we extract a thing X the scanner
00077  * doesn't recognize from 2) a thing Y the scanner does recognize, and 3) if Y
00078  * gets garbage-collected, then X gets freed. If we have code like this:
00079  *
00080  *   void g(JSObject *obj) {
00081  *     jsval x;
00082  *     JS_GetProperty(obj, "x", &x);
00083  *     ... do stuff with x ...
00084  *   }
00085  *
00086  * there's no problem, because the value we've extracted, x, is a jsval, a
00087  * type that the conservative scanner recognizes.
00088  *
00089  * Conservative GC frees us from the obligation to explicitly root the types it
00090  * knows about, but when we work with derived values like |ch|, we must root
00091  * their owners, as the derived value alone won't keep them alive.
00092  *
00093  * A JS::Anchor is a kind of GC root that allows us to keep the owners of
00094  * derived values like |ch| alive throughout the Anchor's lifetime. We could
00095  * fix the above code as follows:
00096  *
00097  *   void f(JSString *str) {
00098  *     JS::Anchor<JSString *> a_str(str);
00099  *     const jschar *ch = JS_GetStringCharsZ(str);
00100  *     ... do stuff with ch, but no uses of str ...;
00101  *   }
00102  *
00103  * This simply ensures that |str| will be live until |a_str| goes out of scope.
00104  * As long as we don't retain a pointer to the string's characters for longer
00105  * than that, we have avoided all garbage collection hazards.
00106  */
00107 template<typename T> class AnchorPermitted;
00108 template<> class AnchorPermitted<JSObject *> { };
00109 template<> class AnchorPermitted<const JSObject *> { };

00111 template<> class AnchorPermitted<const JSFunction *> { };

00113 template<> class AnchorPermitted<const JSString *> { };
00114 template<> class AnchorPermitted<Value> { };
00115 template<> class AnchorPermitted<const JSScript *> { };
00116 template<> class AnchorPermitted<JSScript *> { };

00136 #ifdef __GNUC__
00137 template<typename T>
00138 inline Anchor<T>::~Anchor()
00139 {
00140     /*
00141      * No code is generated for this. But because this is marked 'volatile', G++ will
00142      * assume it has important side-effects, and won't delete it. (G++ never looks at
00143      * the actual text and notices it's empty.) And because we have passed |hold| to
00144      * it, GCC will keep |hold| alive until this point.
00145      *
00146      * The "memory" clobber operand ensures that G++ will not move prior memory
00147      * accesses after the asm --- it's a barrier. Unfortunately, it also means that
00148      * G++ will assume that all memory has changed after the asm, as it would for a
00149      * call to an unknown function. I don't know of a way to avoid that consequence.
00150      */
00151     asm volatile("":: "g" (hold) : "memory");
00152 }
00153 #else
00154 template<typename T>
00155 inline Anchor<T>::~Anchor()
00156 {
00157     /*
00158      * An adequate portable substitute, for non-structure types.
00159      *
00160      * The compiler promises that, by the end of an expression statement, the
00161      * last-stored value to a volatile object is the same as it would be in an
00162      * unoptimized, direct implementation (the "abstract machine" whose behavior the
00163      * language spec describes). However, the compiler is still free to reorder
00164      * non-volatile accesses across this store --- which is what we must prevent. So
00165      * assigning the held value to a volatile variable, as we do here, is not enough.
00166      *
00167      * In our case, however, garbage collection only occurs at function calls, so it
00168      * is sufficient to ensure that the destructor's store isn't moved earlier across
00169      * any function calls that could collect. It is hard to imagine the compiler
00170      * analyzing the program so thoroughly that it could prove that such motion was
00171      * safe. In practice, compilers treat calls to the collector as opaque operations
00172      * --- in particular, as operations which could access volatile variables, across
00173      * which this destructor must not be moved.
00174      *
00175      * ("Objection, your honor!  *Alleged* killer whale!")
00176      *
00177      * The disadvantage of this approach is that it does generate code for the store.
00178      * We do need to use Anchors in some cases where cycles are tight.
00179      *
00180      * NB: there is a Anchor<Value>::~Anchor() specialization below.
00181      */
00182     volatile T sink;
00183     sink = hold;
00184 }
00185 #endif  /* defined(__GNUC__) */

01490 /*
01491  * Methods usually act upon |this| objects only from a single global object and
01492  * compartment.  Sometimes, however, a method must act upon |this| values from
01493  * multiple global objects or compartments.  In such cases the |this| value a
01494  * method might see will be wrapped, such that various access to the object --
01495  * to its class, its private data, its reserved slots, and so on -- will not
01496  * work properly without entering that object's compartment.  This method
01497  * implements a solution to this problem.
01498  *
01499  * To implement a method that accepts |this| values from multiple compartments,
01500  * define two functions.  The first function matches the IsAcceptableThis type
01501  * and indicates whether the provided value is an acceptable |this| for the
01502  * method; it must be a pure function only of its argument.
01503  *
01504  *   static JSClass AnswerClass = { ... };
01505  *
01506  *   static bool
01507  *   IsAnswerObject(const Value &v)
01508  *   {
01509  *       if (!v.isObject())
01510  *           return false;
01511  *       return JS_GetClass(&v.toObject()) == &AnswerClass;
01512  *   }
01513  *
01514  * The second function implements the NativeImpl signature and defines the
01515  * behavior of the method when it is provided an acceptable |this| value.
01516  * Aside from some typing niceties -- see the CallArgs interface for details --
01517  * its interface is the same as that of JSNative.
01518  *
01519  *   static bool
01520  *   answer_getAnswer_impl(JSContext *cx, JS::CallArgs args)
01521  *   {
01522  *       args.rval().setInt32(42);
01523  *       return true;
01524  *   }
01525  *
01526  * The implementation function is guaranteed to be called *only* with a |this|
01527  * value which is considered acceptable.
01528  *
01529  * Now to implement the actual method, write a JSNative that calls the method
01530  * declared below, passing the appropriate template and runtime arguments.
01531  *
01532  *   static JSBool
01533  *   answer_getAnswer(JSContext *cx, unsigned argc, JS::Value *vp)
01534  *   {
01535  *       JS::CallArgs args = JS::CallArgsFromVp(argc, vp);
01536  *       return JS::CallNonGenericMethod<IsAnswerObject, answer_getAnswer_impl>(cx, args);
01537  *   }
01538  *
01539  * Note that, because they are used as template arguments, the predicate
01540  * and implementation functions must have external linkage. (This is
01541  * unfortunate, but GCC wasn't inlining things as one would hope when we
01542  * passed them as function arguments.)
01543  *
01544  * JS::CallNonGenericMethod will test whether |args.thisv()| is acceptable.  If
01545  * it is, it will call the provided implementation function, which will return
01546  * a value and indicate success.  If it is not, it will attempt to unwrap
01547  * |this| and call the implementation function on the unwrapped |this|.  If
01548  * that succeeds, all well and good.  If it doesn't succeed, a TypeError will
01549  * be thrown.
01550  *
01551  * Note: JS::CallNonGenericMethod will only work correctly if it's called in
01552  *       tail position in a JSNative.  Do not call it from any other place.
01553  */
01554 template<IsAcceptableThis Test, NativeImpl Impl>
01555 JS_ALWAYS_INLINE bool
01556 CallNonGenericMethod(JSContext *cx, CallArgs args)
01557 {
01558     const Value &thisv = args.thisv();
01559     if (Test(thisv))
01560         return Impl(cx, args);
01561 
01562     return detail::CallMethodIfWrapped(cx, Test, Impl, args);
01563 }
01564 
01565 JS_ALWAYS_INLINE bool
01566 CallNonGenericMethod(JSContext *cx, IsAcceptableThis Test, NativeImpl Impl, CallArgs args)
01567 {
01568     const Value &thisv = args.thisv();
01569     if (Test(thisv))
01570         return Impl(cx, args);
01571 
01572     return detail::CallMethodIfWrapped(cx, Test, Impl, args);
01573 }

05184 /*
05185  * NB: JS_ExecuteScript and the JS_Evaluate*Script* quadruplets use the obj
05186  * parameter as the initial scope chain header, the 'this' keyword value, and
05187  * the variables object (ECMA parlance for where 'var' and 'function' bind
05188  * names) of the execution context for script.
05189  *
05190  * Using obj as the variables object is problematic if obj's parent (which is
05191  * the scope chain link; see JS_SetParent and JS_NewObject) is not null: in
05192  * this case, variables created by 'var x = 0', e.g., go in obj, but variables
05193  * created by assignment to an unbound id, 'x = 0', go in the last object on
05194  * the scope chain linked by parent.
05195  *
05196  * ECMA calls that last scoping object the "global object", but note that many
05197  * embeddings have several such objects.  ECMA requires that "global code" be
05198  * executed with the variables object equal to this global object.  But these
05199  * JS API entry points provide freedom to execute code against a "sub-global",
05200  * i.e., a parented or scoped object, in which case the variables object will
05201  * differ from the last object on the scope chain, resulting in confusing and
05202  * non-ECMA explicit vs. implicit variable creation.
05203  *
05204  * Caveat embedders: unless you already depend on this buggy variables object
05205  * binding behavior, you should call JS_SetOptions(cx, JSOPTION_VAROBJFIX) or
05206  * JS_SetOptions(cx, JS_GetOptions(cx) | JSOPTION_VAROBJFIX) -- the latter if
05207  * someone may have set other options on cx already -- for each context in the
05208  * application, if you pass parented objects as the obj parameter, or may ever
05209  * pass such objects in the future.
05210  *
05211  * Why a runtime option?  The alternative is to add six or so new API entry
05212  * points with signatures matching the following six, and that doesn't seem
05213  * worth the code bloat cost.  Such new entry points would probably have less
05214  * obvious names, too, so would not tend to be used.  The JS_SetOption call,
05215  * OTOH, can be more easily hacked into existing code that does not depend on
05216  * the bug; such code can continue to use the familiar JS_EvaluateScript,
05217  * etc., entry points.
05218  */

05470  * While getting the length of a string is infallible, getting the chars can
05471  * fail. As indicated by the lack of a JSContext parameter, there are two
05472  * special cases where getting the chars is infallible:
05473  *
05474  * The first case is interned strings, i.e., strings from JS_InternString or
05475  * JSID_TO_STRING(id), using JS_GetInternedStringChars*.
05476  *
