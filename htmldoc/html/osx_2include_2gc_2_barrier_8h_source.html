00016 /*
00017  * A write barrier is a mechanism used by incremental or generation GCs to
00018  * ensure that every value that needs to be marked is marked. In general, the
00019  * write barrier should be invoked whenever a write can cause the set of things
00020  * traced through by the GC to change. This includes:
00021  *   - writes to object properties
00022  *   - writes to array slots
00023  *   - writes to fields like JSObject::shape_ that we trace through
00024  *   - writes to fields in private data, like JSGenerator::obj
00025  *   - writes to non-markable fields like JSObject::private that point to
00026  *     markable data
00027  * The last category is the trickiest. Even though the private pointers does not
00028  * point to a GC thing, changing the private pointer may change the set of
00029  * objects that are traced by the GC. Therefore it needs a write barrier.
00030  *
00031  * Every barriered write should have the following form:
00032  *   <pre-barrier>
00033  *   obj->field = value; // do the actual write
00034  *   <post-barrier>
00035  * The pre-barrier is used for incremental GC and the post-barrier is for
00036  * generational GC.
00037  *
00038  *                               PRE-BARRIER
00039  *
00040  * To understand the pre-barrier, let's consider how incremental GC works. The
00041  * GC itself is divided into "slices". Between each slice, JS code is allowed to
00042  * run. Each slice should be short so that the user doesn't notice the
00043  * interruptions. In our GC, the structure of the slices is as follows:
00044  *
00045  * 1. ... JS work, which leads to a request to do GC ...
00046  * 2. [first GC slice, which performs all root marking and possibly more marking]
00047  * 3. ... more JS work is allowed to run ...
00048  * 4. [GC mark slice, which runs entirely in drainMarkStack]
00049  * 5. ... more JS work ...
00050  * 6. [GC mark slice, which runs entirely in drainMarkStack]
00051  * 7. ... more JS work ...
00052  * 8. [GC marking finishes; sweeping done non-incrementally; GC is done]
00053  * 9. ... JS continues uninterrupted now that GC is finishes ...
00054  *
00055  * Of course, there may be a different number of slices depending on how much
00056  * marking is to be done.
00057  *
00058  * The danger inherent in this scheme is that the JS code in steps 3, 5, and 7
00059  * might change the heap in a way that causes the GC to collect an object that
00060  * is actually reachable. The write barrier prevents this from happening. We use
00061  * a variant of incremental GC called "snapshot at the beginning." This approach
00062  * guarantees the invariant that if an object is reachable in step 2, then we
00063  * will mark it eventually. The name comes from the idea that we take a
00064  * theoretical "snapshot" of all reachable objects in step 2; all objects in
00065  * that snapshot should eventually be marked. (Note that the write barrier
00066  * verifier code takes an actual snapshot.)
00067  *
00068  * The basic correctness invariant of a snapshot-at-the-beginning collector is
00069  * that any object reachable at the end of the GC (step 9) must either:
00070  *   (1) have been reachable at the beginning (step 2) and thus in the snapshot
00071  *   (2) or must have been newly allocated, in steps 3, 5, or 7.
00072  * To deal with case (2), any objects allocated during an incremental GC are
00073  * automatically marked black.
00074  *
00075  * This strategy is actually somewhat conservative: if an object becomes
00076  * unreachable between steps 2 and 8, it would be safe to collect it. We won't,
00077  * mainly for simplicity. (Also, note that the snapshot is entirely
00078  * theoretical. We don't actually do anything special in step 2 that we wouldn't
00079  * do in a non-incremental GC.
00080  *
00081  * It's the pre-barrier's job to maintain the snapshot invariant. Consider the
00082  * write "obj->field = value". Let the prior value of obj->field be
00083  * value0. Since it's possible that value0 may have been what obj->field
00084  * contained in step 2, when the snapshot was taken, the barrier marks
00085  * value0. Note that it only does this if we're in the middle of an incremental
00086  * GC. Since this is rare, the cost of the write barrier is usually just an
00087  * extra branch.
00088  *
00089  * In practice, we implement the pre-barrier differently based on the type of
00090  * value0. E.g., see JSObject::writeBarrierPre, which is used if obj->field is
00091  * a JSObject*. It takes value0 as a parameter.
00092  *
00093  *                                POST-BARRIER
00094  *
00095  * These are not yet implemented. Once we get generational GC, they will allow
00096  * us to keep track of pointers from non-nursery space into the nursery.
00097  *
00098  *                            IMPLEMENTATION DETAILS
00099  *
00100  * Since it would be awkward to change every write to memory into a function
00101  * call, this file contains a bunch of C++ classes and templates that use
00102  * operator overloading to take care of barriers automatically. In many cases,
00103  * all that's necessary to make some field be barriered is to replace
00104  *     Type *field;
00105  * with
00106  *     HeapPtr<Type> field;
00107  * There are also special classes HeapValue and HeapId, which barrier js::Value
00108  * and jsid, respectively.
00109  *
00110  * One additional note: not all object writes need to be barriered. Writes to
00111  * newly allocated objects do not need a pre-barrier.  In these cases, we use
00112  * the "obj->field.init(value)" method instead of "obj->field = value". We use
00113  * the init naming idiom in many places to signify that a field is being
00114  * assigned for the first time.
00115  */

00564 /*
00565  * Incremental GC requires that weak pointers have read barriers. This is mostly
00566  * an issue for empty shapes stored in JSCompartment. The problem happens when,
00567  * during an incremental GC, some JS code stores one of the compartment's empty
00568  * shapes into an object already marked black. Normally, this would not be a
00569  * problem, because the empty shape would have been part of the initial snapshot
00570  * when the GC started. However, since this is a weak pointer, it isn't. So we
00571  * may collect the empty shape even though a live object points to it. To fix
00572  * this, we mark these empty shapes black whenever they get read out.
00573  */

00578 
00579   public:
00580     ReadBarriered() : value(NULL) {}
00581     ReadBarriered(T *value) : value(value) {}
00582 
00583     T *get() const {
00584         if (!value)
00585             return NULL;
00586         T::readBarrier(value);
00587         return value;
00588     }
00589 
00590     operator T*() const { return get(); }
00591 
00592     T &operator*() const { return *get(); }
00593     T *operator->() const { return get(); }
00594 
00595     T **unsafeGet() { return &value; }
00596 
00597     void set(T *v) { value = v; }
00598

00600 };
